# Баг-репорт по микросервису объявлений

В ходе анализа документации и автоматизированного тестирования API были выявлены несоответствия ожидаемому поведению (REST standards).

### 1. Метод GET /item/{id} возвращает массив вместо объекта
* **Описание:** При запросе объявления по уникальному ID сервер возвращает JSON Array (список), содержащий один элемент.
* **Ожидаемое поведение:** Так как ID уникален, сервер должен возвращать JSON Object (словарь) с данными конкретного объявления.
* **Фактический ответ:** `[{ "id": "...", "name": "...", ... }]`
* **Влияние:** Усложняет клиентский код (требуется дополнительная обработка массива).
* **Приоритет:** Medium

### 2. Метод GET /statistic/{id} возвращает массив вместо объекта
* **Описание:** При запросе статистики по уникальному ID объявления сервер возвращает массив объектов статистики.
* **Ожидаемое поведение:** Должен возвращаться один объект статистики `{ "likes": ... }`.
* **Фактический ответ:** `[{ "likes": 10, "viewCount": 5, ... }]`
* **Приоритет:** Medium

### 3. Неструктурированный ответ при создании объявления (POST)
* **Описание:** При успешном создании объявления сервер возвращает JSON с полем `status`, содержащим строку: `"Сохранили объявление - <ID>"`.
* **Ожидаемое поведение:** Сервер должен возвращать JSON-структуру созданного объекта, либо поле `id` должно быть выделено отдельно для удобства парсинга.
* **Фактический ответ:** `{ "status": "Сохранили объявление - 88989d8..." }`
* **Приоритет:** Low (Архитектурное улучшение)

### 4. Ошибка валидации ID (400 вместо 404)
* **Описание:** При передаче в GET-запрос ID, не соответствующего формату UUID (например, "12345"), сервер возвращает ошибку 400 Bad Request.
* **Ожидаемое поведение:** Стандартное поведение для REST — возвращать 404 Not Found, если ресурс не найден, либо более понятное сообщение об ошибке валидации.
* **Приоритет:** Low
